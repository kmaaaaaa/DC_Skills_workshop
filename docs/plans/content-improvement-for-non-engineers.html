コンテンツ改善提案：非エンジニア向け
Skills 作成のための前提知識強化

作成日: 2026-05-21 ステータス: 提案段階（ユーザー承認待ち） 対象ファイル: content/page.md

01目的（What & Why）

現状の content/page.md は 戦略的価値の説明に重きを置いた経営層向けガイド になっており、以下が達成しづらい:

  • 読み終えた人が 「明日から自分で Skills を作れる」状態に到達できない
  • 「Claude / Claude Code / Skills の関係」「フォルダはどこ」「YAML/Markdownの読み方」 といった 最低限の前提 が抜けている
  • 用語（コンテキスト、トークン、MCP、プラグイン、スラッシュコマンド等）の 未定義のまま使用 が散在

本提案では、コンテンツを 「戦略 → 前提知識 → 構造理解 → ハンズオン → 運用」 の学習導線に整え、非エンジニアが自走できる教材へとアップグレードする。

02現状コンテンツの章別評価

章 現状の強み 非エンジニア視点の欠落
s1 戦略的意義 ROI 数値・3社事例が説得力◎ 「Claudeとは／Claude Code との関係」が未定義
s2 機能構造 CLAUDE.md vs Skills 対比、3層構造、キッチンアナロジー YAMLとは何か未説明、フォルダ構造のビジュアル無し、SKILL.md の最小実例なし
s3 判断基準 2×2 マトリクスは秀逸 「自分の業務を当てはめる」演習・境界事例なし
s4 構築プロセス 4ステップ・DBS・skill-creator・コード例あり 環境セットアップ手順なし、トリガー(description)の書き方詳説なし、最小スキル例の段階的解説なし
s5 ユースケース 金融3例 + Before/After 表 職種別の幅出し不足（経理・マーケ・人事・営業）、「明日作れる最小スキル」の具体例なし
s6 リスク管理 落とし穴4件・チェックリスト バージョン管理（Git）への触れ方、チーム共有の3段階（個人/プロジェクト/組織）なし

04改善アプローチ（3案）

ユーザーに選んでいただく必要のある 大方針 を 3 案提示します。

案 A: 前提知識章を新設（章番号を s0 として追加）

  • s1〜s6 はそのまま保持
  • 新 s0「はじめに：環境と前提知識」 を追加
  • K1〜K5、K9 をこの章に集約

メリット: 既存資産を保護、構造変更最小、用語集が早期に登場
デメリット: 章数が増えてボリューム拡大、s1〜s6 内の重複は残る

案 B: 既存6章に補強コンテンツを差し込む（章追加なし）

  • s1〜s6 の構成を保持
  • 各章内に「📝 用語注釈」「🛠️ 実例」「✋ 自分の業務に当てはめる」などの小ブロックを追加
  • 用語集は s6 後ろに付録として配置

メリット: 章構成を変えずに済む（README/CHANGELOG への影響最小）、ナビゲーション安定
デメリット: 補強コンテンツが分散して「全体像」をつかみにくい、ハンズオンへの導線が弱い

案 C: 章構成を再編して「学習導線」を最優先（推奨）

新6章構成:

  1. s1 なぜSkillsか（戦略的意義、現s1から圧縮）
  2. s2 環境とフォルダ構造（新規：K1, K2, K5）
  3. s3 SKILL.mdの読み方・書き方（現s2 + K3, K4, K6）
  4. s4 何をスキル化するか（現s3 + 職種別ユースケース）
  5. s5 作って育てる：実践プロセス（現s4 + K7, K8、ハンズオン強化）
  6. s6 運用・リスク・チーム共有（現s6 + K10）

ユースケース（現s5）は s4 に統合、impact-table は s1 のヒーロー周辺へ移動。用語集は付録として s6 後に配置。

メリット: 学習効果最大、自走できる読者を生み出せる
デメリット: 大規模な書き換え、ナビゲーション全変更、社内既配布版との差分大

05具体的な追加コンテンツリスト

採用案にかかわらず、追加すべきコンテンツの中身 を整理。

5.1 新規ブロック：「環境セットアップ」（K1, K2）

ブロック種別内容
proseClaude（Web）／ Claude Code ／ Claude API の3者比較を1段落で
table3製品 × 用途 × Skills対応 × 起動方法
codeblockClaude Code のインストール1コマンド（Mac/Win）
warn必要環境（Node.js, シェル）

5.2 新規ブロック：「フォルダ構造のビジュアル」（K5）

ツリー図を表現するために、新ブロック種別 tree を新設する必要がある:

~/.claude/
├── skills/              # 個人スキル
│   ├── financial-report/
│   │   ├── SKILL.md     # ← 中核：手順書
│   │   ├── references/  # 参照知識
│   │   └── scripts/     # 補助スクリプト
│   └── weekly-report/
└── CLAUDE.md            # 全体ルール

.claude/                  # プロジェクトスキル（リポジトリ内）
└── skills/
    └── xxx/SKILL.md
推奨：codeblock で代用 — ツリー記号 ├── └── を直接書く方式を採用し、新ブロック追加コストを回避。

5.3 新規ブロック：「Markdown ／ YAML 超入門」（K3, K4）

ブロック種別内容
layers「Markdownの5記法」を5層で（見出し/リスト/コード/強調/リンク）
codeblock---\nname: ...\ndescription: ...\n--- の最小例
pull「YAMLは見出しに使うラベル、本文はMarkdown」と覚える

5.4 新規ブロック：「最小スキル（Hello Skill）」（K6）

  • 「週報フォーマットを整える」スキルを10行で実装
  • 各行に逐行コメントを入れた解説
  • 既存 codeblock 種別で表現可能

5.5 新規ブロック：「description の Good/Bad 例」（K7）

stats 種別を流用、または新規に compare 種別（Bad ↔ Good を並べる）を検討。推奨：既存 stats を流用（before/after を Bad/Good として使う）。

5.6 新規ブロック：「デバッグの初動 3 ステップ」（K8）

既存 steps 種別を流用。

5.7 新規ブロック：「用語集」（K9）

新ブロック種別 glossary を検討、または既存 table で代用（用語 / 一行説明 / 関連項目）。推奨：table で代用。

5.8 新規ブロック：「職種別ユースケース」拡充

既存 usecases 種別を流用、3件 → 6〜9件に拡張:

  • 経理：月次決算サマリー作成
  • マーケ：競合分析レポート
  • 人事：求人原稿レビュー
  • 営業：提案書ドラフト生成
  • 法務：契約書差分チェック
  • IT：障害報告書テンプレート

5.9 「チーム共有の3段階」（K10）

既存 layers 種別を流用:

  • L1 個人スキル（~/.claude/skills/）
  • L2 プロジェクトスキル（.claude/skills/ をリポジトリにコミット）
  • L3 組織配布（プラグインマーケットプレイス／社内 Git）

06新ブロック種別の必要性評価

候補 必要性 代替案 判定
tree (ツリー図) 中 codeblock で代用可 新規不要
compare (Bad/Good) 低 stats で代用可 新規不要
glossary (用語集) 中 table で代用可 新規不要
tutorial (手順 + 期待出力) 低 steps + codeblock で代用 新規不要
結論: ビルドパイプラインに手を入れる必要はなく、既存 19 ブロック種別の組み合わせで実装可能。

09Devil's Advocate（計画時セルフレビュー）

# リスク 対応方針
R1 スコープ Max を選ぶと既存配布版との互換性が崩れる CHANGELOG で BREAKING CHANGE 明記、または v2.0.0 とする
R2 「非エンジニア向け」を意識しすぎて本来の戦略ガイド読者（経営層）の体験を損なう s1（戦略的意義）は短縮しない、追加要素はあくまで補強として配置
R3 用語集の維持コスト（用語が増えるたび更新が必要） s6 後ろの付録に配置し、各章で参照リンクを張る方式に
R4 「Claude Code のインストール」はOS依存・バージョン依存が強く、すぐ陳腐化する URL 直書きはせず「公式ドキュメント参照」の誘導に留める
R5 「Hello Skill」例が読者の業務と乖離すると刺さらない 週報・議事録要約など、ホワイトカラーに共通する題材を選ぶ
R6 ボリューム拡大で読了時間が伸びる（現「READING 約12分」） hero.meta の readingTime を再計算し更新
R7 案 C（章再編）を採用すると過去のリンク（#s1〜#s6）が壊れる 旧 ID は alias として残す or リダイレクト案内を s1 冒頭に
R8 私（メイン）が「非エンジニア向け」を判断するが、実際のターゲット読者像が解像度低い 計画書承認時にユーザーに具体的な読者像（職種・AI習熟度）を確認

10ユーザーに確認したいこと（承認時に決めてほしい）

採用に際して 最低 3 点 の意思決定が必要:

Q1. アプローチ案

案A（前提知識章を新設） / 案B（既存章に補強） / 案C（章再編、推奨）

Q2. 実装スコープ

Min（半日）/ Mid（1日、★推奨）/ Max（2〜3日）

Q3. 想定読者の解像度

  • AI習熟度: 全くの初心者 / Claude Web 利用経験あり / Claude Code 触ったことある
  • 職種範囲: 全職種共通教材 / 特定職種（経理・マーケ等）に最適化
  • 配布形式: 社内勉強会の事前配布資料 / セッション中の投影資料 / 自習用ガイド

Q4. 既存テーマからの逸脱許容度

既存の「戦略 → 機能 → 判断 → 構築 → 事例 → リスク」順序を 死守する / 再編許容

11完了条件（実装フェーズに進む場合）

承認後の実装計画は別途作成しますが、最低限の合格基準:

  • 非エンジニアが章末まで読めば、SKILL.md を自力で1つ書ける
  • 章中に登場する専門用語が全て初出時に定義されている（または用語集にリンク）
  • 環境セットアップからHello Skill 完成までの最短経路が示されている
  • 既存の戦略的価値訴求（s1）の説得力を損なわない
  • npm run build が成功し、視覚的に崩れがない

  • ダークテーマ版 (claude-skills-dark.html) への反映
  • インタラクティブ要素（クイズ、進捗表示）
  • 動画・図解アニメーション
  • 印刷向けPDF化
